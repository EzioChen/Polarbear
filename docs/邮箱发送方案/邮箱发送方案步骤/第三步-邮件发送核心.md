# 第三步：邮件发送核心

## 阶段概述

本阶段实现邮件发送的核心功能，包括邮件构建、SMTP连接管理、发送执行和重试机制。这是整个邮件服务最关键的部分，直接决定了邮件能否成功发送。

---

## 任务一：MailComposer 邮件构建器

### 任务描述

实现邮件内容构建功能，将用户输入的邮件选项转换为符合 Nodemailer 要求的邮件格式。

### 任务目标

1. 实现邮件内容组装
2. 实现发件人信息处理
3. 实现附件处理
4. 实现邮件头处理

### 前置条件

- 已完成配置管理系统
- 已了解 Nodemailer 的邮件格式要求

### 详细步骤

#### 1.1 邮件内容组装

**输入处理：**
- 接收 SendEmailOptions 对象
- 验证必填字段（to、subject，以及 text/html 至少一个）
- 处理收件人格式（字符串转数组）

**发件人处理：**
- 从配置中获取发件人信息
- 格式化为 "名称 <邮箱>" 或仅邮箱
- 示例：`"PolarBear 发布系统 <noreply@example.com>"`

**内容处理：**
- 同时支持 text 和 html 内容
- 如只提供一种格式，自动转换另一种（可选）
- HTML 内容需要完整的文档结构

#### 1.2 附件处理

**附件来源支持：**
- 本地文件路径（path）
- 文件内容（Buffer/string/Stream）
- URL 地址（href）

**附件预处理：**
- 读取本地文件内容为 Buffer
- 从 URL 下载附件内容
- 自动检测 MIME 类型（根据文件扩展名）
- 验证文件大小（不超过限制）

**附件格式转换：**
将自定义 Attachment 格式转换为 Nodemailer 格式：
```typescript
{
  filename: string;
  content?: Buffer;
  path?: string;
  contentType?: string;
  contentDisposition?: 'attachment' | 'inline';
  cid?: string;
}
```

#### 1.3 邮件头处理

**标准头处理：**
- 优先级映射（high/normal/low 到 X-Priority）
- Message-ID 生成（可选）
- Date 头自动添加

**自定义头处理：**
- 合并用户提供的 headers
- 验证头名称格式
- 防止注入攻击（过滤换行符）

#### 1.4 输出格式

**MailOptions 结构：**
```typescript
{
  from: string;           // 发件人
  to: string[];           // 收件人
  cc?: string[];          // 抄送
  bcc?: string[];         // 密送
  subject: string;        // 主题
  text?: string;          // 纯文本
  html?: string;          // HTML
  attachments?: Attachment[];  // 附件
  headers?: Record<string, string>;  // 自定义头
}
```

### 预期成果

- MailComposer 类实现完整
- 邮件内容正确组装
- 附件处理功能正常

### 验收标准

- [ ] 邮件内容组装正确
- [ ] 发件人信息格式正确
- [ ] 附件处理支持多种来源

## 任务二：MailSender 发送执行器

### 任务描述

实现邮件发送的核心执行逻辑，包括 SMTP 连接创建、邮件发送和连接关闭。

### 任务目标

1. 实现 SMTP 传输器创建
2. 实现邮件发送执行
3. 实现连接生命周期管理
4. 实现发送结果格式化

### 前置条件

- 已完成配置管理
- 已了解 Nodemailer 的 Transporter API

### 详细步骤

#### 2.1 传输器创建

**创建流程：**
1. 从 ConfigManager 获取配置
2. 构建 Nodemailer 配置对象
3. 调用 createTransport 创建传输器

**Nodemailer 配置映射：**
```typescript
{
  host: config.smtp.host,
  port: config.smtp.port,
  secure: config.smtp.secure,
  tls: config.smtp.tls,
  auth: {
    user: config.auth.user,
    pass: config.auth.pass,  // 从 SecretStorage 获取
    type: config.auth.type
  },
  connectionTimeout: config.connection.timeout,
  greetingTimeout: config.connection.greetingTimeout,
  socketTimeout: config.connection.socketTimeout
}
```

#### 2.2 连接生命周期管理

**设计原则（单次发送模式）：**
- 每次发送独立创建连接
- 发送完成后立即关闭连接
- 不支持连接池

**生命周期流程：**
```
发送开始
  ↓
创建传输器（建立连接）
  ↓
发送邮件
  ↓
关闭传输器（断开连接）
  ↓
返回结果
```

**错误处理：**
- 创建连接失败：抛出 CONNECTION_FAILED
- 认证失败：抛出 AUTH_FAILED
- 发送超时：抛出 SEND_TIMEOUT

#### 2.3 邮件发送执行

**发送流程：**
1. 调用 transporter.sendMail(mailOptions)
2. 记录发送开始时间
3. 等待发送完成
4. 计算发送耗时

**超时处理：**
- 使用 Promise.race 实现超时控制
- 超时时间从配置读取
- 超时后强制关闭连接

#### 2.4 结果格式化

**成功结果：**
- success: true
- messageId: SMTP 返回的消息 ID
- timestamp: ISO 8601 格式时间
- response: SMTP 响应字符串
- accepted: 成功接收的收件人列表
- rejected: 被拒绝的收件人列表
- duration: 发送耗时（毫秒）

**失败结果：**
- success: false
- error: EmailError 对象
- timestamp: 失败时间
- duration: 已耗时（毫秒）

**错误包装：**
- 捕获 Nodemailer 错误
- 根据错误类型映射到 EmailErrorCode
- 保留原始错误信息

### 预期成果

- MailSender 类实现完整
- SMTP 连接管理正确
- 发送结果格式化正确

## 任务三：重试机制实现

### 任务描述

实现发送失败后的自动重试机制，提高发送成功率。

### 任务目标

1. 实现重试策略配置
2. 实现指数退避算法
3. 实现可重试错误判断
4. 实现重试执行流程

### 前置条件

- 已完成 MailSender 基础发送功能
- 已了解重试配置参数

### 详细步骤

#### 3.1 重试策略配置

**配置参数：**
- enabled: 是否启用重试（默认 true）
- maxRetries: 最大重试次数（默认 3）
- retryDelay: 初始重试延迟（默认 5000ms）
- exponentialBackoff: 是否使用指数退避（默认 true）

#### 3.2 指数退避算法

**算法实现：**
```typescript
function calculateDelay(attempt: number, baseDelay: number, useExponential: boolean): number {
  if (useExponential) {
    return baseDelay * Math.pow(2, attempt - 1);
  }
  return baseDelay;
}
```

**延迟计算示例：**
- 第1次重试：5000ms
- 第2次重试：10000ms
- 第3次重试：20000ms

#### 3.3 可重试错误判断

**可重试错误类型：**
- CONNECTION_FAILED: 连接失败（可能是网络问题）
- CONNECTION_TIMEOUT: 连接超时
- SEND_TIMEOUT: 发送超时
- TLS_ERROR: TLS 错误（可能是临时问题）

**不可重试错误类型：**
- AUTH_INVALID_CREDENTIALS: 认证失败（配置错误）
- RECIPIENT_INVALID: 收件人无效（输入错误）
- MESSAGE_TOO_LARGE: 邮件过大（内容错误）

**判断逻辑：**
```typescript
function shouldRetry(errorCode: EmailErrorCode): boolean {
  const retryableCodes = [
    EmailErrorCode.CONNECTION_FAILED,
    EmailErrorCode.CONNECTION_TIMEOUT,
    EmailErrorCode.SEND_TIMEOUT,
    EmailErrorCode.TLS_ERROR
  ];
  return retryableCodes.includes(errorCode);
}
```

#### 3.4 重试执行流程

**重试流程：**
```
执行发送
  ↓
成功？→ 返回结果
  ↓ 失败
判断可重试？
  ↓ 否
抛出错误
  ↓ 是
达到最大重试次数？
  ↓ 是
抛出 RETRY_EXHAUSTED 错误
  ↓ 否
计算延迟时间
  ↓
等待延迟
  ↓
重试计数 +1
  ↓
重新执行发送
```

**重试计数：**
- 记录当前重试次数
- 达到 maxRetries 后停止重试
- 每次重试使用新的连接

### 预期成果

- 重试机制完整实现
- 指数退避算法正确
- 可重试错误判断准确

### 验收标准

- [ ] 指数退避计算正确
- [ ] 可重试/不可重试错误判断正确

## 任务四：EmailService 核心服务

### 任务描述

实现 EmailService 核心服务类，整合配置管理、邮件构建和发送功能，提供统一的邮件发送接口。

### 任务目标

1. 实现单例模式

### 前置条件

- 已完成 ConfigManager、MailComposer、MailSender
- 已了解单例模式实现

### 详细步骤

#### 4.1 单例模式实现

**实现方式：**
- 私有静态 instance 属性
- 私有构造函数
- 公共静态 getInstance() 方法

**线程安全：**
- JavaScript 单线程，无需额外同步
- 确保异步初始化完成后再使用

#### 4.2 服务初始化

**初始化流程：**
1. 创建 ConfigManager 实例
2. 加载配置文件
3. 验证配置
4. 创建 MailComposer 实例
5. 创建 MailSender 实例
6. 标记初始化完成

**错误处理：**
- 配置加载失败：抛出 CONFIG_NOT_FOUND 或 CONFIG_INVALID
- 配置验证失败：抛出 CONFIG_MISSING_FIELD
- 记录初始化日志

**初始化状态：**
- 提供 `isInitialized()` 方法查询状态
- 未初始化时调用 sendEmail 抛出 SERVICE_NOT_INITIALIZED

#### 4.3 邮件发送接口

**接口实现：**
```typescript
async sendEmail(options: SendEmailOptions): Promise<SendResult>
```

**发送流程：**
1. 检查初始化状态
2. 使用 MailComposer 构建邮件
3. 使用 MailSender 发送邮件
4. 返回发送结果
5. 记录发送日志

**前置验证：**
- 验证必填字段
- 验证收件人邮箱格式
- 验证附件存在性

#### 4.4 配置验证接口

**接口实现：**
```typescript
async validateConfig(): Promise<ValidationResult>
```

**功能：**
- 调用 ConfigManager 验证配置
- 额外验证 SMTP 连接参数
- 返回详细验证结果

#### 4.5 连接测试接口

**接口实现：**
```typescript
async testConnection(): Promise<ConnectionResult>
```

**测试流程：**
1. 检查初始化状态
2. 记录开始时间
3. 创建传输器并验证连接
4. 关闭连接
5. 计算响应时间
6. 返回连接结果

**测试内容：**
- 能否连接到 SMTP 服务器
- 能否完成认证
- 响应时间
